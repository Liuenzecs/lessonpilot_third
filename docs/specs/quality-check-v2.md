# 教学质量检查 2.0 规格

## 目标

在导出前不仅判断“能不能导出”，还判断“能不能交、能不能教”。

## 检查维度

- 必填信息：标题、学科、年级、课时、核心 section
- 确认状态：默认只导出 confirmed 内容，pending section 需要提示
- 教学目标：是否具体、可观察、可评价
- 重难点：是否区分重点和难点，是否说明真实学习障碍
- 教学过程：是否至少 3 个环节，是否有教师活动、学生活动、设计意图和合理时长
- 一致性：目标是否在教学过程中被承接，测评是否覆盖目标
- 学案质量：学习目标是否学生可读，题目是否有答案和解析
- 泛化风险：识别“提高综合素养”等空泛套话

## 响应语义

沿用 `ready / needs_fixes / blocked`：

- `blocked`：关键元信息缺失或文档几乎为空
- `needs_fixes`：可导出，但正式提交或上课前建议修复
- `ready`：主要风险通过

响应继续包含 `issues / warnings / suggestions`，并增加 `alignment_map` 用于展示目标、过程、评价之间的对应关系。

## 可执行修复

Phase 11 起，部分质量问题可转成待确认修订：

- `POST /api/v1/documents/{document_id}/quality-fix`
- 请求包含问题所在 `section`、`message` 和 `suggestion`
- 服务端只处理明确可自动辅助的高频问题：目标具体化、过程承接目标、评价补齐、学生活动可执行化、重难点进入过程
- 修订后的 section 必须保持 `pending`，由老师确认后才进入最终稿
- 无法识别的问题返回清晰错误，不做静默改写
